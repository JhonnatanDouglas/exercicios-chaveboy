Modulo 4 - Backend Express - Sprint 3

Controle de Projetos - KenzieVelopers

Crie uma API para desenvolvedores, informacoes extras e projetos. O foco e relacionamento 1:1 e 1:N no PostgreSQL.

PostgreSQLrelacionamentosExpressTypeScript

Dependencias principais

  • express e express-async-errors: servidor e tratamento de erros async.
  • typescript e ts-node-dev: desenvolvimento em TypeScript.
  • pg e pg-format: PostgreSQL e queries dinamicas.
  • dotenv: variaveis de ambiente.
  • jest, supertest e ts-jest: testes de integracao, se forem exigidos.

Tabelas

  • developers: id, name, email unico.
  • developerInfos: id, developerSince, preferredOS, developerId unico.
  • projects: id, name, description, repository, startDate, endDate, developerId.

preferredOS deve usar enum com Windows, Linux e MacOS.

Relacionamentos

  • Um developer tem no maximo uma developerInfo.
  • Ao deletar developer, sua developerInfo deve ser deletada junto.
  • Um developer pode ter muitos projects.
  • Ao deletar developer, projects ficam com developerId = NULL.

Endpoints

MetodoRotaFuncao
POST/developersCriar developer
GET/developers/:idBuscar developer com info
PATCH/developers/:idAtualizar developer
DELETE/developers/:idDeletar developer
POST/developers/:id/infosCriar info extra
POST/projectsCriar projeto
GET/projects/:idBuscar projeto
PATCH/projects/:idAtualizar projeto

Ordem de montagem

  1. Crie sql/createTables.sql com enum OS, tabelas e relacionamentos.
  2. Configure .env, conexao PostgreSQL e inicializacao do banco.
  3. Monte a base Express com app.ts, server.ts e middleware global de erros.
  4. Crie a estrutura routers, controllers, services, middlewares e interfaces.
  5. Implemente /developers: criar, listar por id, atualizar e deletar.
  6. Adicione middlewares de email duplicado e developer inexistente.
  7. Implemente /developers/:id/infos validando OS e info duplicada.
  8. Implemente /projects e valide developerId quando enviado.
  9. Implemente GET/PATCH de projects validando id inexistente.
  10. Revise retornos, status codes e efeitos dos relacionamentos no banco.

Regras de aceite

  • Email de developer deve ser unico.
  • Developer inexistente deve retornar 404 em GET, PATCH, DELETE e criacao de info.
  • Cada developer pode ter apenas uma info extra.
  • preferredOS aceita somente Windows, Linux ou MacOS.
  • Projeto deve aceitar developerId existente ou ficar sem developer.
  • Ao deletar developer, info some junto e projects ficam com developerId nulo.
